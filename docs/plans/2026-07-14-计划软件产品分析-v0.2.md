# 不秃｜计划软件产品分析文档

> 版本：v0.2  
> 日期：2026-07-14  
> 状态：已完成首轮产品定义，待进入 PRD  
> 首版平台：Web
> 产品名称：不秃

## 一、产品定义

这是一款面向个人的长期计划与日常执行工具。它帮助用户快速记录想法，把任意计划继续拆分为子计划，再通过拖拽将计划项安排到某月、某周、某日或具体时间段。每一天的行动都可以追溯到它所属的长期计划，底层完成情况也会自动汇总到上层。

产品首先服务于创建者本人，重点兼顾两种体验：记录和排期必须足够简单；长期规划、延期处理与复盘又必须足够深入。它不要求用户先建立完整的人生计划体系，也不强迫每项任务选择分类、优先级或上级目标。用户可以先把想法放入收集箱，之后逐步整理。

核心对象统一为“计划项”。任何计划项都可以拥有子项、计划周期、执行日期、状态、进度和复盘。计划的父子关系表示“这件事服务于哪个目标”，日期与时间表示“准备什么时候执行”，两者相互独立。

产品不把拖延视为失败。逾期事项不会被系统擅自滚动，而是进入“待重新安排”，帮助用户选择今天完成、重新排期、拆分任务或放弃，并保留延期历史供复盘使用。

### 一句话定位

> 像待办工具一样轻松记录，像成长系统一样连接长期目标、每日行动与复盘。

## 二、目标用户与核心闭环

首版产品以创建者本人为第一用户：有长期目标，也愿意制定计划，但容易因为计划过重、任务过大或状态变化而拖延。用户需要的不是更严格的监督工具，而是一套能快速记录、灵活调整，并帮助自己持续复盘的个人系统。

### 核心使用闭环

1. **捕捉**：通过全局快捷入口，仅填写标题，将想法放入收集箱；在某个计划内创建时，自动继承当前计划的归属。
2. **目标规划**：把计划项归入某个大计划，或继续拆分子项。计划项可选择人生方向、阶段目标、年度目标、项目、普通任务等角色标签，但不强制填写。
3. **执行排期**：把计划项拖到某月、某周或某日。排期越具体，展示位置越精确；计划项的长期归属不会因拖拽而改变。
4. **行动**：当天任务分为“未定时间”和“具体时间段”，可在待办、进行中、已完成、已放弃之间切换。
5. **调整**：逾期事项进入“待重新安排”，用户可以选择今天做、移到某个周期、指定日期、拆分或放弃。系统保留排期变化和延期原因。
6. **复盘**：系统自动汇总完成、延期和时间分布数据，通过少量问题引导日、周、月复盘；底层完成情况同步汇总到上层计划。

核心价值循环：

```text
捕捉 → 目标规划 → 执行排期 → 行动 → 调整 → 复盘 → 改进下一轮计划
```

## 三、核心概念与业务规则

产品采用“目标线 + 执行线”的双线模型，两条线共享同一个核心对象：计划项。

### 3.1 目标线

目标线回答“为什么做、要达成什么”。任何计划项都能拥有子项，形成自由层级的目标树。用户可以给计划项添加可选角色，例如人生方向、阶段目标、年度目标、项目或普通任务。角色只用于理解、筛选和模板推荐，不限制层级，也不要求逐层创建。

```text
成为独立开发者                [人生方向]
└─ 发布自己的第一款产品        [阶段目标]
   └─ 完成计划软件网页版       [项目]
      └─ 完成产品分析文档       [普通任务]
```

### 3.2 执行线

执行线回答“什么时候推进”。计划项可以先安排到某个月，再逐渐明确到某周、某日和具体时间段。系统只保存当前有效的计划周期或执行时间，并根据日期自动判断它属于哪个年、月和周，避免多个时间字段互相冲突。

```text
7 月 → 本周 → 今天 → 14:00—16:00
```

### 3.3 核心规则

- 父子关系与执行时间相互独立。
- 拖拽排期只修改时间，不改变计划项的目标归属。
- 未设置日期的计划项保留在收集箱或待安排清单。
- 只设置日期的计划项显示在当天“未定时间”区域。
- 设置开始时间和预计时长后，计划项显示为小时日历中的时间块。
- 任意计划项都可以继续拆分，不区分固定的“计划”和“任务”类型。
- 叶子计划项通过状态表示完成情况；有子项的计划默认按子项完成比例汇总进度。
- 上层计划可以切换为手动进度；子项权重暂不进入首版。
- 年、月、周、日默认是系统生成的时间视图，同时允许填写该周期的主题、重点和复盘。
- 月度主题属于目标表达；月、周、日中的具体事项属于执行安排。

## 四、延期、重复与复盘机制

### 4.1 延期与重新安排

产品将延期视为计划与现实之间的反馈，而不是用户失败。计划项超过预定日期且尚未完成时，不会自动移动到今天，也不会一直隐藏在过去，而是进入“待重新安排”清单。

用户可以快速选择：

- 今天完成
- 移到本周或下周
- 选择新的日期
- 将任务拆成更小的子项
- 标记为已放弃

每次重新排期都会保留原日期、新日期和调整时间。延期原因可以选填，例如任务过大、时间不足、精力不足、计划变化或等待他人。连续延期时，系统温和提示用户考虑拆分、重新评估价值或降低本次目标，不使用惩罚式文案。

### 4.2 简单重复

首版支持每天、每周和每月三种简单重复规则，并允许设置开始和结束日期。重复计划作为模板，按周期生成独立执行记录，因此某一次可以单独完成、延期或跳过，而不影响整个重复计划。

复杂工作日规则、节假日顺延，以及“仅本次、从本次以后、全部重复项”等高级修改范围留到后续版本。

### 4.3 双复盘体系

- **周期复盘**：日、周、月、年，用于观察完成情况、延期、时间投入和状态变化。
- **计划复盘**：重要计划完成或放弃时，总结成果、偏差和经验。

首版重点实现日、周、月复盘以及大计划的简短完成总结。系统先自动生成客观数据摘要，再提供少量可跳过的引导问题和自由书写区域。

AI 总结暂不接入。后续可以根据子计划完成情况、延期记录及各级复盘生成大计划总结草稿，用户编辑确认后再保存。

## 五、信息架构与核心页面

### 5.1 今日首页

每天打开产品后的默认页面。左侧展示精简目标树和当前计划的长期归属；中间展示今日重点、未定时间任务以及小时日历；右侧展示收集箱和待重新安排事项。用户可以直接完成、开始、延期或拖动任务。

### 5.2 目标规划

以树状结构展示所有计划项。用户可以创建人生方向、阶段目标、年度目标、项目和普通任务，并自由添加子项。点击节点后，在详情区域查看说明、子项、进度、计划周期、执行安排和完成总结。

### 5.3 计划工作台

负责跨周期排期。用户可以在月、周、日视图间切换，将收集箱或目标树中的计划项拖到相应时间。进入日视图后，可把未定时间任务拖入小时日历，并调整开始时间和预计时长。批量选择计划项后，也可以统一移动到某月、周或日期。

### 5.4 复盘中心

集中进入日、周、月复盘，并查看系统生成的完成数、延期次数、时间分布和未完成事项。历史复盘按照时间归档；重要计划的完成总结保存在对应计划详情中。

### 5.5 数据与设置

管理简单重复规则、显示偏好、Markdown 与 JSON 导出、JSON 备份恢复。首版无需注册登录，数据保存在当前浏览器。页面需要明确提示本地数据风险，并提供便捷备份入口。

全局顶部始终保留快速记录框，让用户在任何页面都能将想法送入收集箱。

## 六、MVP 范围与版本规划

首版目标不是一次实现完整的人生管理系统，而是验证“目标规划与日常执行打通”是否真的能帮助用户持续使用。

### 6.1 MVP 必须完成

- 全局快速记录与收集箱
- 统一计划项的创建、编辑、删除和父子拆分
- 可选计划角色标签
- 目标树与计划详情
- 月、周、日时间视图
- 单项与批量拖拽排期
- 日视图的未定时间区和小时日历
- 待办、进行中、已完成、已放弃四种状态
- “重要”及每天建议 1—3 项“今日重点”
- 子项完成率自动汇总与手动进度
- 逾期检测和待重新安排清单
- 延期历史与可选延期原因
- 每天、每周、每月简单重复
- 日、周、月复盘和计划完成总结
- Markdown 导出
- JSON 完整备份与恢复
- 浏览器本地数据保存

### 6.2 第二阶段

- 注册登录与多设备云同步
- 自定义标签和人生领域
- 年度复盘、统计仪表盘和高级筛选
- 提醒与浏览器通知
- PDF 复盘报告
- 更完整的重复规则
- AI 辅助生成计划复盘草稿

### 6.3 后续阶段

- App 与 Windows 客户端
- 离线和云端双向同步
- 习惯追踪、搜索、附件和模板
- Notion、Obsidian 等外部集成
- 开放 API
- 共享与协作计划

### 6.4 MVP 验收闭环

用户能完成：

```text
快速记录 → 目标拆解 → 拖拽排期 → 当日执行或延期
→ 日/周复盘 → 查看上层进度 → 导出并恢复数据
```

## 七、非功能要求、数据安全与成功指标

### 7.1 体验要求

- 首次打开后，无需注册即可开始记录。
- 常用操作尽量在一至两步内完成。
- 快速记录只要求填写标题。
- 拖拽后立即显示结果，并提供撤销入口。
- 删除、批量移动、放弃大计划等高影响操作需要确认。
- 页面刷新后保留当前数据和主要界面状态。
- 延期、逾期和复盘文案保持中性、鼓励调整，不制造羞耻感。
- 键盘操作应覆盖快速记录、完成任务和日期移动等高频动作。
- 电脑端优先保证拖拽和小时日历体验；手机浏览器至少支持快速记录、查看今日任务、修改状态和简单排期。

### 7.2 数据安全

数据默认保存在浏览器本地，不上传服务器。系统需要明确说明：清除浏览器数据、重装系统或更换设备可能造成数据丢失。

首版必须支持 JSON 完整备份和恢复，并在用户长期未备份时温和提醒。导入前进行格式校验，恢复操作不得静默覆盖现有数据，应提供合并、替换或取消选项；执行替换前再次确认。

### 7.3 成功指标

首版主要验证个人实际使用价值，不追求注册量。建议观察：

- 是否能够连续使用四周
- 每周是否至少完成一次计划和复盘闭环
- 收集箱中的事项能否被及时规划或排期
- 延期事项能否被重新决策，而不是持续堆积
- 今日重点的完成情况
- 上层计划是否因为日常行动得到真实推进
- 使用过程中是否逐渐减少对其他计划软件的依赖

最核心的成功标准是：产品能让用户更容易重新开始，而不是因为一次延期就放弃整套计划。

## 八、关键用户故事与验收场景

### 8.1 快速捕捉

作为用户，我可以在任意页面只输入标题创建计划项。提交后，它立即出现在收集箱中，不要求选择日期、角色或上级计划。

### 8.2 目标规划

作为用户，我可以把计划项拖入另一个计划项，或在计划详情中创建子项。调整父子关系后，执行日期保持不变；上级计划能够汇总子项进度。

### 8.3 逐步排期

作为用户，我可以将计划项安排到某月、某周或某日。确定具体日期后，系统自动将其展示在对应的年、月、周视图中，不产生重复计划项。改变排期不会改变目标归属。

### 8.4 小时日历

作为用户，我可以把当天“未定时间”的任务拖入小时日历。系统根据落点设置开始时间和默认时长；我可以移动或拉伸时间块，也可以将它拖回未定时间区。

### 8.5 延期处理

作为用户，当任务逾期时，我会在“待重新安排”中看到它。我可以重新排期、拆分或放弃。系统保留原日期与调整历史，不会自动将任务移动到今天。

### 8.6 重复执行

作为用户，我可以设置每天、每周或每月重复。每次执行产生独立记录，因此某一次延期、跳过或完成，不影响其他周期。

### 8.7 周期复盘

作为用户，我可以进入日、周、月复盘。系统先展示完成数、延期事项和时间投入，再提供少量引导问题。保存后，复盘可以在历史记录中查看。

### 8.8 数据恢复

作为用户，我可以导出完整 JSON 备份。在另一个空白环境中导入后，计划树、排期、状态、延期历史、重复规则和复盘都能正确恢复。

## 九、已确认的关键决策

| 主题 | 决策 |
|---|---|
| 首批用户 | 先服务创建者本人，再逐步产品化 |
| 核心取向 | 简单快速记录与拖拽排期，同时支持复盘和长期成长 |
| 核心对象 | 所有内容统一为“计划项”，任意计划项均可继续拆分 |
| 创建入口 | 全局快速记录进入收集箱；计划内创建自动继承归属 |
| 目标与时间 | 父子归属和执行排期相互独立 |
| 时间视图 | 年、月、周、日自动生成，同时允许填写周期主题与复盘 |
| 状态 | 待办、进行中、已完成、已放弃 |
| 聚焦 | 不设高、中、低优先级；使用“重要”和“今日重点” |
| 延期 | 进入待重新安排，不自动滚动到今天 |
| 进度 | 默认按子项等权汇总，可切换手动；权重后续再做 |
| 重复 | MVP 支持每天、每周、每月 |
| 标签 | MVP 仅保留计划角色；自定义标签和人生领域后续实现 |
| AI | MVP 不接入，第二阶段考虑 AI 复盘草稿 |
| 数据 | MVP 本地优先，支持 Markdown 导出与 JSON 备份恢复 |
| 云端 | 第二阶段加入登录和多设备同步 |

## 十、暂不纳入 MVP

- AI 生成与模型调用
- 云端账号和多设备同步
- 子计划进度权重
- 自定义标签和固定人生领域
- 高级重复规则和节假日处理
- PDF 报告
- 协作与共享
- App 与 Windows 客户端

上述能力应在数据模型和架构中保留扩展空间，但不应阻碍首版核心闭环的实现。
