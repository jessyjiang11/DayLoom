# DayLoom / 不秃 v0.4 架构与 UI 设计

## 1. 目标

把 v0.3 可点击原型升级为真正可使用的在线 Web MVP：用户可以注册登录，选择示例或空白工作区，保存自己的目标、任务、排期、标签、复盘和最小个人资料，并在多个在线设备之间实时看到更新。

v0.4 同时完成一次视觉重构：从“展示型卡片界面”改为克制、安静、适合长期使用的“编辑型计划本”。

## 2. 范围

### v0.4 包含

- 邮箱 + 密码注册、邮箱验证、登录、退出和找回密码。
- 首次进入选择“使用示例”或“空白开始”。
- 最小个人资料：昵称、头像、时区、每周开始日、界面偏好。
- 目标树、任务编辑、月/周/日排期、拖拽改期、标签、今日重点和复盘。
- 云端自动保存、同步状态提示、最近成功数据缓存。
- 两个在线设备之间的 Realtime 更新。
- 数据库 RLS 隔离、输入校验、错误和空状态。
- 响应式 Web 与 GitHub Pages 部署。

### v0.4 不包含

- AI 生成复盘。
- 完整离线编辑与冲突合并。
- 团队共享、协作空间和公开计划。
- 社交登录、付费订阅、推送通知。
- 原生 iOS 或 Windows 客户端。

## 4. 前端结构

```text
web/
├─ src/
│  ├─ app/              路由、Providers、应用入口
│  ├─ components/       Button、Field、Dialog、SyncStatus 等基础组件
│  ├─ features/
│  │  ├─ auth/          登录、注册、找回密码
│  │  ├─ onboarding/    示例/空白选择、初始资料
│  │  ├─ today/         今日重点、时间轴、未定时任务
│  │  ├─ goals/         目标树和详情
│  │  ├─ planner/       月、周、日排期与拖拽
│  │  ├─ reviews/       日/周/月复盘
│  │  └─ settings/      个人资料和偏好
│  ├─ lib/              Supabase、日期、缓存、校验
│  ├─ repositories/     数据访问与实时订阅
│  ├─ styles/           tokens、base、layout、components
│  ├─ test/             测试初始化与 fixtures
│  └─ types/            数据库和领域类型
└─ vite.config.ts
```

- React Router 管理页面和认证保护。
- TanStack Query 管理云端数据、加载、错误、缓存和重新获取。
- Supabase JS 负责账户、CRUD 与 Realtime。
- dnd-kit 提供鼠标、触摸和键盘可访问拖拽。
- Zod 在表单提交和数据库响应边界做运行时校验。
- 页面短期 UI 状态使用组件状态或 reducer，不引入全局状态库。

## 5. 数据模型

所有标识使用 UUID；所有时间戳使用 `timestamptz`；数据库字段使用小写 `snake_case`。每张用户业务表都有 `user_id`、`created_at`、`updated_at`，可同步实体增加 `version` 与 `deleted_at`。

### profiles

- `user_id uuid primary key references auth.users(id) on delete cascade`
- `display_name text`
- `avatar_url text null`
- `timezone text default 'Asia/Shanghai'`
- `week_starts_on smallint default 1 check (week_starts_on between 0 and 6)`
- `preferences jsonb default '{}'`
- `onboarding_mode text null check (onboarding_mode in ('sample','blank'))`

### items

统一保存方向、阶段目标、项目与普通任务。

- `id uuid primary key`
- `user_id uuid not null`
- `parent_id uuid null references items(id)`
- `kind text check (kind in ('direction','goal','project','task'))`
- `title text not null`
- `description text default ''`
- `status text check (status in ('todo','doing','done','abandoned'))`
- `sort_order numeric`
- `is_important boolean`
- `is_focus boolean`
- `schedule_granularity text null check (schedule_granularity in ('month','week','day','time'))`
- `schedule_date date null`
- `schedule_start_time time null`
- `schedule_period_start date null`
- `schedule_period_end date null`
- `duration_minutes smallint null`
- `version bigint default 1`
- `deleted_at timestamptz null`

### schedule_changes

保存重新排期历史：`item_id`、`from_schedule jsonb`、`to_schedule jsonb`、`reason`、`changed_at`。

### tags / item_tags

- `tags` 保存用户自定义标签名与颜色。
- `item_tags` 使用 `(item_id, tag_id)` 联合主键。
- 两张表都包含 `user_id`，便于 RLS 直接判断归属。

### period_notes

保存本周重点、本周一句话和以后可能扩展的月度意图：`period_type`、`period_start`、`focus_text`、`note_text`。

### reviews

保存日/周/月复盘：`period_type`、`period_start`、`summary`、`went_well`、`blocked`、`next_step`。

## 6. 数据安全

- 所有用户表启用并强制 RLS。
- `authenticated` 用户只能读取、插入、更新、删除 `user_id = (select auth.uid())` 的行。
- `insert` 同时使用 `with check`，防止伪造其他用户 ID。
- `user_id`、`parent_id`、排期字段和所有外键建立索引。
- 前端只使用 Supabase publishable/anon key；`service_role` 永不进入 Web、iOS 或 GitHub。
- SQL migration 是权限真相来源，不依赖 Dashboard 手工记忆。
- 头像上传限制 MIME、大小和用户目录；v0.4 可先使用首字母头像，延后 Storage 上传。

## 7. 同步模型

- 登录后先显示最近成功缓存，再请求云端最新数据。
- 未登录时不加载业务数据。
- 写操作采用乐观更新；失败则回滚并显示可重试提示。
- Realtime 订阅当前用户的 items、tags、item_tags、period_notes 和 reviews。
- 收到远端更新后按 `id + version + updated_at` 合并到 Query Cache。
- Realtime 断开时显示“正在重新连接”，恢复后强制重新获取。
- 离线时界面进入只读模式；不建立离线写队列。

## 8. 视觉方向：编辑型计划本

### 原则

- 白色画布，几乎不使用渐变。
- 依靠排版、留白和 1px 分隔线建立层级，而不是大量悬浮卡片。
- 去掉每页重复的口号式大标题；页面首先回答“现在是什么时间、我能做什么”。
- 阴影仅用于模态编辑器、拖拽浮层和必要的悬浮层。
- 圆角缩减为 4/8/12px 三档，避免所有元素胶囊化。
- 动效只表达状态变化，时长 120–220ms，并尊重 `prefers-reduced-motion`。

### 视觉令牌

- 字体：`Microsoft YaHei Light`, `Microsoft YaHei`, `微软雅黑`, sans-serif。
- 主文字：`#24313a`；次文字：`#6f7a82`；边线：`#e1e5e8`。
- 蓝：`#587895`，用于当前导航与选中。
- 绿：`#5f8974`，用于完成与成长。
- 橙：`#c4864d`，用于提醒和待整理。
- 红：`#b86b68`，仅用于逾期、删除和今日重点提示。
- 背景：`#ffffff`；次级区：`#f7f8f8`。

### 核心页面

- 登录：单列窄表单，没有营销插画；清楚说明数据会同步到个人账户。
- 今日：日期、同步状态、今日重点、小时日历和右侧未定时列表。
- 目标：左侧大纲树，右侧编辑详情；新增顶层目标与子计划明确分离。
- 计划：月/周/日使用真实日历结构；粗排期区与具体日期区明确分层。
- 设置：个人资料、周起始日、时区、数据导出和退出登录。

## 9. 首次使用流程

```text
注册 → 邮箱验证 → 设置昵称/时区/每周开始日
     → 选择“使用示例”或“空白开始” → 今日页
```

示例数据由一个受 RLS 保护的数据库函数一次性创建；重复调用不会重复生成。

## 10. 非功能要求

- 375px、768px、1280px 宽度下可用。
- 主要操作支持键盘，表单和对话框满足基本 WCAG AA 对比度与焦点要求。
- 暖缓存首屏目标小于 2 秒；保存操作立即反馈状态。
- 所有云端失败提供重试，不用静默失败。
- 早期目标为个人项目规模（约 1,000 用户），不为假设的大规模提前引入微服务。
- 成本以 Supabase Free + GitHub Pages 为目标；超过免费范围前必须明确提示。

## 11. 失败模式

| 场景 | 用户表现 | 处理 |
|---|---|---|
| 网络断开 | 显示缓存，只读 | 监听网络恢复并重新获取 |
| 登录过期 | 保存被拒绝 | 刷新会话；失败则回登录页 |
| 保存失败 | 乐观更新回滚 | 显示原因与重试按钮 |
| Realtime 断开 | 设备间暂不同步 | 显示重连状态，恢复后全量刷新 |
| Supabase 暂停 | 无法登录/读取 | 提供明确错误，不误称“没有数据” |
| 两端同时修改 | 后到写入覆盖 | v0.4 使用版本与更新时间提示；完整冲突 UI 后续实现 |

## 12. 验收标准

- 两个不同账户无法通过前端或直接 API 读取对方数据。
- 同一账户在两个浏览器中修改任务，另一端能实时更新。
- 刷新、退出再登录或换设备后数据仍存在。
- 首次注册可以选择示例或空白，示例不会重复生成。
- 页面不再依赖大面积渐变、浓阴影和口号式大标题。
- v0.3 `prototype/` 仍可独立打开和运行测试。
