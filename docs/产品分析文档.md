> **版本**: v0.1 | **日期**: 2026-07-14 | **状态**: 产品定义阶段

## 一、核心理念

### 1.1 这个产品的本质是什么？

这是一个 **「计划链」管理系统**。它的核心命题是：**让每一天的行动，都能追溯到人生愿景。**

绝大多数人的计划是断裂的——年度目标存在于某个文档，周计划在笔记软件，每日待办在另一个 App。三套系统互不关联，导致：

- 做着做着就忘了年初的目标是什么
- 每天很忙但不知道在忙什么
- 复盘时发现琐事消耗了所有时间，重要目标毫无进展

本产品的核心价值主张：**建立一个从「人生愿景 → 长期目标 → 年度 OKR → 月度里程碑 → 周聚焦 → 日行动」的完整链条**，并且让这条链上的信息可以双向流动——分解向下、汇总向上、任意节点可跳转查看上下文。

### 1.2 解决的痛点

| 痛点 | 现状 | 本产品方案 |
|---|---|---|
| **层级断裂** | 人生计划、年计划、日计划分散在不同工具中 | 六层计划在一个统一模型内，自动建立父子关联 |
| **战略漂移** | 年度目标从未分解为可执行的日常行动 | 强制要求分解链路，每个子级计划必须关联上级 |
| **复盘困难** | 不同层级复盘需要手动翻找多个文档 | 专注模式直接展示当前层级+上下级关联，一键进入复盘 |
| **进度不可见** | 不知道年度目标完成了多少，感受不到积累 | 进度自动从底层向上汇总，可视化展示完成百分比 |
| **反馈缺失** | 不知道哪类计划总是完不成，无法归因 | 统计跨层级完成率、时间分配，识别瓶颈领域 |

### 1.3 与竞品的差异化

| 竞品 | 定位 | 本产品差异 |
|---|---|---|
| **Notion** | 万能文档/数据库，需自行搭建 | 原生的六层计划模型，开箱即用，无需设计模板 |
| **Todoist** | 任务管理（项目/任务） | 向上延伸到战略层，让待办有「来处」 |
| **飞书多维表格** | 结构化数据管理 | 有层级逻辑和自动汇总，而非纯数据表 |
| **滴答清单** | 日程+待办，轻量 GTD | 增加中长期计划维度，不只是「今天做什么」 |
| **Vision / 目标管理类** | OKR 目标追踪 | 向下打通到每周每日，让目标落在行动上 |
| **Things 3** | Apple 生态优雅待办 | 跨平台 + 更长周期的计划管理 |

**一句话差异化定位**：唯一一款以「计划层级链」为第一公民、从人生愿景管到每日行动的闭环工具。

## 二、层级关系模型

### 2.1 六层计划体系

`
人生计划 (Life Plan)          ← 10年以上，愿景/价值观
    └── 长期计划 (Long-term)   ← 3-5年，关键目标
        └── 年度计划 (Annual)  ← 1年，OKR 式关键结果
            └── 月度计划 (Monthly) ← 1个月，里程碑/聚焦项
                └── 周计划 (Weekly)  ← 1周，行动聚焦
                    └── 日计划 (Daily)   ← 1天，具体待办
`

### 2.2 关系规则

- **一对多父子关系**：一个上级可拆解为多个下级；一个下级必须有且仅有一个直接上级
- **允许越级引用**：日计划可直接关联年度计划中的关键结果（但主上级仍是周计划）
- **允许独立存在**：用户可从任一层级开始使用（如只做日计划），但上层级需要提示「该计划未被任何上级目标关联」
- **软关联 vs 硬关联**：初期采用软关联（可断开，仅提示），成熟后可选硬关联模式（强制对齐）

### 2.3 自上而下分解 vs 自下而上汇总

| 方向 | 机制 | 用户动作 |
|---|---|---|
| **自上而下（分解）** | 创建下级计划时，从上级目标中「拆解」出子任务 | 点击「分解」按钮，系统自动带入上级上下文 |
| **自下而上（汇总）** | 日计划完成自动更新周进度；周进度→月进度→… | 无需手动操作，系统自动计算完成率并向上传递 |
| **双向校准** | 如果自上而下的分解在执行中被证明不合理，允许从底层反向调整上级目标 | 月度复盘时，根据实际进展调整年度 KR |

**核心设计原则**：
1. 上层定义「方向」（做什么、为什么）
2. 下层定义「执行」（怎么做、做多少）
3. 进度自动向上流动，调整可自上而下也可自下而上

## 四、核心功能清单

### 4.1 MVP 核心功能（P0）

| 功能 | 说明 | 优先级 |
|---|---|---|
| **六层计划 CRUD** | 每一层级完整的创建、编辑、删除、查看 | P0 |
| **计划分解** | 从上级计划一键「分解」创建下级计划，自动带入上下文 | P0 |
| **父子关联与跳转** | 任意计划可查看上级/下级列表，支持点击跳转 | P0 |
| **进度自动汇总** | 日→周→月→年→长期的完成率自动向上传递 | P0 |
| **今日视图** | 聚合展示今日待办、本周聚焦、月度目标 | P0 |
| **多层级导航** | 侧边栏/面包屑支持六层快速切换 | P0 |

## 五、典型用户场景与流程

### 5.1 场景一：年初规划（自上而下分解）

`mermaid
flowchart TD
  A["📝 打开年度计划页面"] --> B["点击「新建年度计划」"]
  B --> C["选择关联的长期计划"]
  C --> D["填写年度目标 + 关键结果 KR"]
  D --> E["按季度拆分里程碑"]
  E --> F["为每个季度创建月度计划草稿"]
  F --> G["发布计划，开始执行"]
`

**用户价值**：一次性完成年度→月度分解，后续月计划只需微调，不偏离方向。

### 5.2 场景二：每日行动（执行层）

`mermaid
flowchart TD
  A["🌅 早晨打开「今日视图」"] --> B["查看今日待办 + 本周聚焦"]
  B --> C["点击任务可上溯到关联的年度 KR"]
  C --> D["执行任务，标记完成/推迟"]
  D --> E["🌙 晚间填写复盘：今日反思 + 精力评分"]
  E --> F["系统自动更新周/月进度"]
`

**用户价值**：每天知道自己「为什么做这件事」，以及它对年度目标的贡献度。

### 5.3 场景三：周复盘（检视调整）

`mermaid
flowchart TD
  A["📆 周末打开周复盘视图"] --> B["系统展示本周完成率、未完成项"]
  B --> C["逐项复盘：哪些完成了、为什么有些没完成"]
  C --> D["查看月度目标进度，判断是否需要调整"]
  D --> E["规划下周聚焦 + 分解下周任务"]
  E --> F["保存周复盘笔记"]
`

**用户价值**：在「失控」发生前及时校准，而不是到了年底才发现目标没达成。

### 5.4 场景四：俯瞰全景（回顾与调整）

`mermaid
flowchart TD
  A["🗺️ 打开「全景视图」"] --> B["展示人生计划→长期→年度的树状结构"]
  B --> C["每个节点显示完成进度百分比"]
  C --> D["点击进入任意节点查看详情"]
  D --> E["发现某领域长期停滞 → 调整优先级或放弃"]
`

**用户价值**：一眼看清全局，识别「虚假忙碌」和「被遗忘的目标」。

### 5.5 场景五：越级关联（灵活映射）

`mermaid
flowchart TD
  A["🔄 日常做任务时"] --> B["创建一个今日任务"]
  B --> C["关联到周计划（默认父级）"]
  C --> D["额外关联到年度计划中的某个 KR"]
  D --> E["完成此任务同时推动周进度和年度 KR 进度"]
`

**用户价值**：有些日常任务不通过常规层级分解，但仍对年度目标有贡献，允许灵活映射。

## 六、技术选型建议

### 6.1 核心原则

- **本地优先 (Local-first)**：用户数据所有权第一，离线可用
- **渐进增强**：先用本地存储做出可用产品，再逐步添加云同步
- **轻量架构**：单人开发或小团队可维护，避免过度工程化

### 6.2 推荐方案

| 层级 | 技术选项 | 理由 |
|---|---|---|
| **前端框架** | **React 18 + TypeScript** | 生态最丰富、UI 库选择多、社区活跃 |
| **构建工具** | **Vite** | 极速 HMR、开箱即用 TS 支持 |
| **UI 组件库** | **Tailwind CSS + shadcn/ui** | 高度可定制、设计质量好、无 vendor lock-in |
| **状态管理** | **Zustand** | 轻量、TS 友好、适合中等复杂度应用 |
| **路由** | **React Router v6** | 标准方案，支持嵌套路由（对应层级嵌套） |
| **本地存储 MVP** | **Dexie.js (IndexedDB)** | 浏览器端结构化存储，支持复杂查询，无需后端 |
| **云端同步 P1** | **Supabase (PostgreSQL + Auth)** | 开源 BaaS、自带实时同步、RLS 行级安全 |
| **Markdown 编辑** | **Tiptap / Milkdown** | 现代 WYSIWYG+Markdown 编辑器 |
| **图表可视化** | **Recharts / ECharts** | 进度仪表盘、甘特图 |
| **日历组件** | **FullCalendar** | 成熟稳定的日历视图组件 |
| **部署 MVP** | **Vercel / Cloudflare Pages** | 零配置部署、全球 CDN |
| **桌面端 P2** | **Tauri** | 比 Electron 轻量，Rust 后端性能好 |

### 6.3 架构演进路径

`
Phase 1 (MVP):       纯前端 SPA + IndexedDB 本地存储
                      ↓
Phase 2 (同步):      + Supabase 后端，实现多设备云同步
                      ↓
Phase 3 (桌面端):    + Tauri 包装，提供原生体验和系统通知
                      ↓
Phase 4 (协作):      + 团队空间、共享计划、权限管理
`

## 附录：关键设计决策记录

### A. 为什么用六层而不是四层或八层？

- **人生计划** 必须有——这是「为什么做」的根
- **长期计划（3-5年）** 是年度到人生之间的必要缓冲——年度跨度太短，人生跨度太长
- **年度→月度→周→日** 是自然时间单位的级联，符合人类认知和复盘习惯
- 暂不设「季度计划」层级，通过年度计划的 quarterly_milestones 字段承载，避免层级过多

### B. 为什么让日计划关联年度 KR（越级）？

实际生活中，很多日常行动并不经过「月度→周」的正式分解流程，但确实在为年度目标做贡献。允许越级关联可以在不增加层级复杂度的前提下，保证所有行动都能被追溯到年度目标。

### C. 数量约束建议

| 层级 | 建议数量上限 | 理由 |
|---|---|---|
| 人生计划 | 1 | 人生愿景应聚焦 |
| 长期计划 | 3-7 | 对应各人生领域的关键方向 |
| 年度计划 | 5-10 | 每年不宜同时追逐过多目标 |
| 月度计划 | 3-5 聚焦项 | 超出此数容易分散 |
| 周计划 | 1-3 聚焦领域 | 一周精力有限 |
| 日计划 | 5-8 任务 | 每日核心任务不宜过多 |

系统不做硬性限制，但在 UI 上做温和提示（如超过建议数量时显示温和警告）。
